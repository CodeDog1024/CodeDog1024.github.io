操作和访问数据库

前边学习了如何连接数据库，当获取数据库连接后就可以操作数据库了，Java 提供了三个接口用于操作数据库：

  1. Statement：用于执行静态 SQL 语句并返回结果
  2. PreparedStatement：SQL 语句被预编译并保存在此对象中，可以使用此对象多次高效的执行该语句
  3. CallableStatement：用于执行 SQL 存储过程

Statement

Statement 用于执行静态 SQL 语句并返回结果，是因为其无法动态的拼接 SQL，如果使用字符串拼接的方式动态拼接 SQL，会存在 SQL 注入的问题

SQL注入：利用某些系统的可输入控件，且该系统没有对输入内容进行充分校验，在输入的数据中注入非法的 SQL 语句段或者命令，从而利用 SQL 引擎进行恶意行为的做法。

因此，PreparedStatement 诞生了，PreparedStatement 提供了对应的方法来设置 SQL 参数，从而有效地避免了因为字符串拼接而产生的 SQL 注入问题。

PreparedStatement

前边了解到 Statement 可以用来执行 SQL 语句，但是有一些弊端：

  1. 如果 SQL 包含参数，执行通过字符串拼接的方式拼接 SQL
  2. 存在 SQL 注入问题

因此 PreparedStatement 诞生了，它是 Statement 的子接口，在其基础上新增了一功能，并弥补了弊端。

ResultSet

当我们执行查询操作时，会返回一个结果集 ResultSet，其中保存了 SQL 查询出来的数据，为了方便查看数据，我们需要将数据进行转换，这时候就要引入 ORM 编程思想。

ORM（Object Relational Mapping）意思是：

  1. 一个表对应一个类
  2. 一条记录对应一个对象
  3. 一个字段对那个一个属性

而此时就涉及到了 MySQL 中字段的类型与 Java 中数据类型的映射关系，不同版本的数据库驱动映射关系可能不一致。

这是，我们可以通过 ResultSet 来获取元数据，在元数据中，保存了具体的列数和列名。

新的问题出现了，我们虽然可以获取列的个数和值，但是我们不知道该列对应哪个属性，这个值该往哪个属性设置呢？

前边提到了，类名和属性名是由对应关系的，如：id → id、username → username、create_time → createTime。因此我们可以先获取列名，然后通过反射的方式，使用列名来获取属性名，再通过反射设置值到属性中。

此时，执行程序时，控制台会报错：java.lang.NoSuchFieldException: create_time，因为我们需要把 create_time 转为 createTime 才能获取属性，此时有两种办法：

  1. 在 SQL 中给 create_time 起别名，但是需要使用 getColumnLabel 来获取列名，而不是 getColumnName，前者可以获取别名

  2. 自已编写一个方法用于将 xxx_xxx 转为 xxxXxx。

但是我们不能只针对于 tb_user 来进行查询，因此返回值不能仅限于 User 类，所以我们可以通过泛型 + 反射来动态创建对象。
