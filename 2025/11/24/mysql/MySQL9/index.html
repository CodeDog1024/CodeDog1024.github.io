MySQL SQL 优化

|浏览量:

插入优化

普通插入

插入一条数据

批量插入

当需要插入多条数据时，不建议使用多个插入语句，而是使用批量插入语句，将多个数据放在一个 INSERT 语句中。

但是尽管批量插入可以插入多条数据，但是不建议超过 1000 条，若超过 1000 条数据，可以将数据以 1000 条为一组分批插入。

事务提交

MySQL 默认是自动提交事务，所以当执行多个插入语句时，每次插入都会提交一次事务，导致性能较差。我们可以手动开启事务，然后执行全部插入语句，最后再手动提交事务。这样子多次插入只需要一次事务提交，性能会提高很多。

主键顺序

若主键存在顺序，则插入数据时，尽量按照主键的顺序插入，这样子性能会提高很多。

大批量插入

尽管批量插入可以插入多条数据，但是性能仍有限，若插入数据量达到几十万乃至百万，则可以使用 MySQL 数据库提供的 LOAD 指令来进行批量插入，这样子性能会提高很多。

操作步骤：

  1. 连接数据库时添加参数：mysql --local-infile -u root -p
  2. 修改全局配置：SET GLOBAL local_infile = 1;
  3. 使用 LOAD DATA 命令进行批量插入：LOAD DATA LOCAL INFILE '/data.txt' INTO TABLE tb_user FIELDS TERMINATED BY ',' LINES TERMINATED BY '\n';

主键优化

数据组织方式

在 InnoDB 存储引擎中，数据是按照主键的顺序进行存储的，这种存储方式的表称为索引组织表（IOT），因此在设计表结构时，尽量选择顺序增长的主键（如自增主键、时间戳等），这样子插入数据时，能够减少数据页的分裂，提高插入性能。

页分裂

前边学习过，InnoDB 存储引擎中，数据是以页为单位进行存储的，默认页大小为 16KB。单行记录的大小不能超过页大小的一半，因此，每页至少存储两条记录，每条记录的大小不超过 8KB。如果超出限制，则会发生行溢出。

当主键被顺序插入时，数据会被顺序地插入到页的末尾，当页被填满时，则会创建新的页，并把数据添加到新的页中。

而当主键被乱序插入时，数据会按照顺序插入到页中指定的位置，但是当该页已经填满时，无法插入该数据，则会造成页分裂，从而造成性能损耗。

页合并

当删除数据时，若某个页中的数据被删除后，剩余数据量小于指定的阈值时，则会发生页合并，将该页的数据与相邻的页合并，从而释放该页。

该阈值可以通过 MERGE_THRESHOLD 参数进行设置，默认值为 50%。

主键设计原则

  1. 尽量降低逐渐的长度。
  2. 尽量选择自增主键，或者其他递增主键，尽量按照顺序插入数据。
  3. 尽量不要使用 UUID 等自然主键，因为 UUID 是随机的，插入数据时会导致页分裂，影响插入性能。且 UUID 较长，会增加索引的大小，影响查询性能。

排序优化

在 MySQL 中，排序操作分为两种：

  1. Using filesort：表示使用文件排序，通常是将数据先存储到内存中进行排序，若内存不足，则会将数据写入到磁盘文件中进行排序，性能较差。
  2. Using index：表示使用索引排序，通常是通过索引来进行排序，性能较好。

分组优化

分页优化

使用 limit 进行分页查询时，若页数较大，则会导致性能较差。因为 MySQL 需要扫描前面的数据，然后再返回指定页的数据。

优化方法：

  1. 使用索引覆盖，只查询索引字段，不查询其他字段
  2. 通过子查询的方式，先查询出主键，然后再通过主键查询数据

COUNT 优化

在 MyISAM 中，COUNT 并不需要实时计算，MyISAM 会维护一个计数器，直接返回计数器的值，因此 COUNT 查询非常快。

在 InnoDB 中，COUNT 需要实时计算，因此 COUNT 慢。

在 MySQL 中，目前没有比较好的解决方案。但是我们可以通过其他技术来替代 COUNT，比如自己在 Redis 中维护一个计数器，每次插入时，将计数器加 1，每次删除时，将计数器减 1。

COUNT 分类

分类 描述
COUNT(*) 统计行数，不取值，直接累加行数，性能最高
COUNT(主键) 统计行数，取值，性能一般
COUNT(column) 统计非 NULL 值，性能较差，除非有 NOT NULL 约束
COUNT(1) 统计行数，不取值，直接累加行数，性能约等于 COUNT(*)

更新优化

在事务中，如果执行了更新语句，则会对目标行上锁，直到事务结束（提交或回滚），再次期间，其他会话无法对改行进行操作

  1. 如果更新条件中包含索引，则为行级锁，只锁定符合条件的行，其他会话仍可操作其他行
  2. 如果更新条件中不包含索引，则为表级锁，锁定整张表，其他会话无法操作该表

因此，在执行更新操作时，更新条件中一定要包含索引。避免使用表级锁。

总结

  1. 插入优化：批量插入，手动提交事务，主键按顺序插入，大批量数据使用 LOAD DATA 命令
  2. 主键优化：主键长度要短，尽可能按顺序插入，避免页分裂
  3. 排序优化：创建合适的索引，满足最左前缀法则，尽可能使用 Using index，避免 Using filesort
  4. 分组优化：创建合适的索引，满足最左前缀法则
  5. 分页优化：使用索引覆盖，子查询主键，再通过主键查询数据
  6. COUNT 优化：使用 COUNT(*) 或 COUNT(主键)，避免 COUNT(column)
  7. 更新优化：更新条件中包含索引，避免使用表级锁
